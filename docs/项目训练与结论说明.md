# 贵州省逐日降雨预测项目训练与结论说明

## 1. 项目目标

本项目基于 [20152024.xlsx](C:/Coding/260219_ML-rainfall-time-prediction/20152024.xlsx) 中 2015-2024 年贵州省逐日降雨数据，完成多城市逐日降雨预测。项目最终目标不是只输出一个单值，而是同时得到：

- 点预测结果：回答“明天大概下多少雨”
- 区间预测结果：回答“明天的降雨大概率落在哪个范围内”

最终实践表明，这两个目标不适合强行用一个模型同时做到最优，因此项目最终采用“双模型协同”方案。

## 2. 数据与任务定义

核心数据读取逻辑在 [data.py](C:/Coding/260219_ML-rainfall-time-prediction/data.py)。这里做了三件关键事情：

- 读取 Excel，并兼容 `date`、`Unnamed: 0` 以及 Excel 序列号日期
- 增加季节性时间特征 `doy_sin` / `doy_cos`
- 按年份切分训练、验证、测试集

项目统一采用以下任务设定：

- 输入窗口长度 `lookback_L = 60`
- 预测步长 `horizon_H = 1`
- 训练集：2015-2022
- 验证集：2023
- 测试集：2024

这部分训练主流程由 [scripts/train.py](C:/Coding/260219_ML-rainfall-time-prediction/scripts/train.py) 负责，具体包括数据切窗、模型构建、损失计算、验证早停、测试集评估和预测文件导出。

## 3. 为什么没有直接采用 `tdsan-master`

[simple-predic-20260308/tdsan-master](C:/Coding/260219_ML-rainfall-time-prediction/simple-predic-20260308/tdsan-master) 中的原始代码更像一个单变量电池时序样例项目，不是为贵州多城市降雨数据直接设计的。它的数据目录 [TDSAN/TDSAN/data](C:/Coding/260219_ML-rainfall-time-prediction/simple-predic-20260308/tdsan-master/TDSAN/TDSAN/data) 中主要是 `B0005/B0006/B0007/B0018` 这类单列序列文件，任务形式是固定窗口预测下一步，和本项目的多城市逐日降雨问题不一致。

因此，本项目没有直接把 `tdsan-master` 当训练主线，而是只借鉴“时序编码器替换”的思路，在当前降雨主线上实现了更适配的 BiMamba 版本。

## 4. 最终选定的模型

最终统一骨架是 [models/simple_bimamba.py](C:/Coding/260219_ML-rainfall-time-prediction/models/simple_bimamba.py) 中的 `SimpleBiMambaForecast`。它的结构比较清晰：

- `in_proj`：把输入特征映射到 `d_model`
- `pos_emb`：增加可学习的位置编码
- `encoder`：堆叠多层 `ExBiMamba`
- `pool_proj`：融合最后时刻特征和全局平均特征
- `head`：根据任务模式输出点预测或分位数预测
- `domain_head`：保留了与现有训练框架兼容的域头接口

对应代码里最关键的任务分支是：

```python
if task_mode == "point":
    self.head = nn.Linear(d_model, H)
elif task_mode == "quantile":
    self.head = nn.Linear(d_model, H * len(self.quantiles))
```

这也是后面能实现“同一骨架、不同训练目标”的基础。

### 4.1 点预测最终模型

最终点预测模型配置为 [gz_rain_allcities_point_bimamba_final.yaml](C:/Coding/260219_ML-rainfall-time-prediction/configs/gz_rain_allcities_point_bimamba_final.yaml)：

- `model.name: simple_bimamba`
- `task.mode: point`
- `target_transform: log1p`
- `feature_mode: all`
- `neighbor_mode: all`
- `lookback_L: 60`

这套配置的目标很明确：尽量降低 `MAE/RMSE`，给出一个更准的单值预测。

### 4.2 区间预测最终模型

最终区间预测模型配置为 [gz_rain_allcities_quantile_bimamba_final.yaml](C:/Coding/260219_ML-rainfall-time-prediction/configs/gz_rain_allcities_quantile_bimamba_final.yaml)：

- `model.name: simple_bimamba`
- `task.mode: quantile`
- `quantiles: [0.05, 0.5, 0.95]`
- `target_transform: asinh`
- `rain_weighting.weights: [1.0, 1.6, 3.8, 6.5, 10.0]`

这套配置的目标不是单纯最小化点误差，而是兼顾：

- `PICP`
- `PINAW`
- `AQL`

也就是尽量让区间既能覆盖真实值，又不要宽得失去意义。

## 5. 训练流程

完整训练入口是 [scripts/train.py](C:/Coding/260219_ML-rainfall-time-prediction/scripts/train.py)。主流程可以概括为：

1. 读取 [20152024.xlsx](C:/Coding/260219_ML-rainfall-time-prediction/20152024.xlsx)
2. 通过 [data.py](C:/Coding/260219_ML-rainfall-time-prediction/data.py) 增加时间特征并按年份切分
3. 用 [dataset.py](C:/Coding/260219_ML-rainfall-time-prediction/dataset.py) 做滑动窗口
4. 根据配置创建 `SimpleBiMambaForecast` 或 `DADeformMamba`
5. 点预测任务使用点损失，区间任务使用分位数损失
6. 在验证集上早停，保存最佳模型
7. 在测试集上输出 `metrics.json` 与 `predictions.csv`

项目里针对强降雨样本还加入了按阈值加权的逻辑，关键函数在 [scripts/train.py](C:/Coding/260219_ML-rainfall-time-prediction/scripts/train.py) 中的 `rain_sample_weight`。测试阶段还会统计暴雨分层指标，关键函数是同文件中的 `rain_band_metrics` 与 `summarize_predictions`。

## 7. 调参路径与关键经验

本项目不是一开始就直接得到最终方案，而是经过了多轮试验。下面用贵阳市说明关键调整逻辑。

### 7.1 初始简化版的问题

初始简化版配置是 [gz_rain_guiyang_simple_bimamba.yaml](C:/Coding/260219_ML-rainfall-time-prediction/configs/gz_rain_guiyang_simple_bimamba.yaml)，结果见 [runs/gz_rain_guiyang_simple_bimamba/贵阳市/metrics.json](C:/Coding/260219_ML-rainfall-time-prediction/runs/gz_rain_guiyang_simple_bimamba/贵阳市/metrics.json)：

- `RMSE = 10.6686`
- `MAE = 4.1352`
- `PICP = 0.5761`
- `PINAW = 0.2124`

问题是：区间偏窄但覆盖率太低，很多强降雨样本被明显低估。

### 7.2 点预测与区间预测开始拆开

点预测优化版结果见 [runs/gz_rain_guiyang_point_bimamba_opt/贵阳市/metrics.json](C:/Coding/260219_ML-rainfall-time-prediction/runs/gz_rain_guiyang_point_bimamba_opt/贵阳市/metrics.json)：

- `RMSE = 9.7647`
- `MAE = 4.1369`

区间优化版结果见 [runs/gz_rain_guiyang_quantile_bimamba_opt/贵阳市/metrics.json](C:/Coding/260219_ML-rainfall-time-prediction/runs/gz_rain_guiyang_quantile_bimamba_opt/贵阳市/metrics.json)：

- `RMSE = 9.8581`
- `MAE = 4.0478`
- `PICP = 0.7908`
- `PINAW = 0.3061`

这一步已经显现出一个重要规律：点预测指标和区间指标的最优配置不一致。

### 7.3 暴雨权重增强与折中

为了改善强降雨低估，项目尝试了：

- 重加权版 [gz_rain_guiyang_quantile_bimamba_heavyrain_v3.yaml](C:/Coding/260219_ML-rainfall-time-prediction/configs/gz_rain_guiyang_quantile_bimamba_heavyrain_v3.yaml)
- 平衡权重版 [gz_rain_guiyang_quantile_bimamba_balanced_v4.yaml](C:/Coding/260219_ML-rainfall-time-prediction/configs/gz_rain_guiyang_quantile_bimamba_balanced_v4.yaml)

对应结果文件分别是：

- [v3 metrics](C:/Coding/260219_ML-rainfall-time-prediction/runs/gz_rain_guiyang_quantile_bimamba_heavyrain_v3/贵阳市/metrics.json)
- [v4 metrics](C:/Coding/260219_ML-rainfall-time-prediction/runs/gz_rain_guiyang_quantile_bimamba_balanced_v4/贵阳市/metrics.json)

经验是：

- 暴雨权重加得太重，会改善 `>=25mm`、`>=50mm` 样本，但会拉高整体 `MAE`
- 适度平衡权重更稳，但仍然不能同时做到点预测最优和区间最优

### 7.4 `asinh` 变换对区间预测更有利

在 [gz_rain_guiyang_quantile_bimamba_asinh_v5.yaml](C:/Coding/260219_ML-rainfall-time-prediction/configs/gz_rain_guiyang_quantile_bimamba_asinh_v5.yaml) 中，区间模型改为 `target_transform: asinh`。结果见 [runs/gz_rain_guiyang_quantile_bimamba_asinh_v5/贵阳市/metrics.json](C:/Coding/260219_ML-rainfall-time-prediction/runs/gz_rain_guiyang_quantile_bimamba_asinh_v5/贵阳市/metrics.json)：

- `RMSE = 9.6606`
- `MAE = 4.6926`
- `PICP = 0.8791`
- `PINAW = 0.2531`

它的特点很鲜明：

- 总体 `MAE` 不是最佳
- 但区间覆盖率与区间宽度的平衡更好
- 对强降雨样本更有帮助

因此，这一版最终被保留为区间预测主方案。

### 7.5 更复杂骨架并没有带来更好结果

项目还测试了 [gz_rain_guiyang_dadeformmamba_asinh_v6.yaml](C:/Coding/260219_ML-rainfall-time-prediction/configs/gz_rain_guiyang_dadeformmamba_asinh_v6.yaml)，结果见 [runs/gz_rain_guiyang_dadeformmamba_asinh_v6/贵阳市/metrics.json](C:/Coding/260219_ML-rainfall-time-prediction/runs/gz_rain_guiyang_dadeformmamba_asinh_v6/贵阳市/metrics.json)：

- `RMSE = 9.8487`
- `MAE = 4.9312`
- `PICP = 0.8235`
- `PINAW = 0.2866`

这说明在当前任务规模和数据设定下，更复杂的 `DADeformMamba` 并没有超过 `SimpleBiMamba`。因此，最终模型没有继续增加结构复杂度，而是保留了更稳定、可解释性更好的 `SimpleBiMamba`。

### 7.6 为什么最终没有采用 `DA_DeformMamba_GZRain`

项目中原本就有更复杂的降雨主线模型，即 [models/da_deformmamba.py](C:/Coding/260219_ML-rainfall-time-prediction/models/da_deformmamba.py) 里的 `DADeformMamba`。它相比 `SimpleBiMamba` 多了几层能力：

- 时间维 `DeformBlock`，用于做时间对齐
- `ConBiMambaEncoder`，用于长依赖建模
- 可选空间分支 `spatial_deform_enabled`
- 域分类头，兼容域对抗训练

从结构上看，它确实更强，也更接近项目早期的主线设想；但最终没有把它作为省域最终方案，原因主要有三点。

第一，当前任务目标已经收敛为“多城市逐日预测 + 点值/区间双任务分开建模”，而不是继续追求一个结构上尽可能复杂的统一模型。对这个目标来说，`SimpleBiMamba` 已经能够稳定完成输入编码、长依赖提取和多任务输出，额外加入时序可变形注意力、空间分支和域对抗接口，并没有在最终指标上体现出确定性的收益。

第二，实验结果没有支持它成为最终模型。以贵阳市对照实验为例，[v5](C:/Coding/260219_ML-rainfall-time-prediction/runs/gz_rain_guiyang_quantile_bimamba_asinh_v5/贵阳市/metrics.json) 的 `SimpleBiMamba + asinh` 结果为：

- `RMSE = 9.6606`
- `MAE = 4.6926`
- `PICP = 0.8791`
- `PINAW = 0.2531`

而 [v6](C:/Coding/260219_ML-rainfall-time-prediction/runs/gz_rain_guiyang_dadeformmamba_asinh_v6/贵阳市/metrics.json) 的 `DADeformMamba + asinh` 结果为：

- `RMSE = 9.8487`
- `MAE = 4.9312`
- `PICP = 0.8235`
- `PINAW = 0.2866`

也就是说，在这组直接对照中，`DADeformMamba` 的总体误差、区间覆盖率和区间宽度都没有超过 `SimpleBiMamba`。

第三，从工程角度看，`DADeformMamba` 的训练成本、调参复杂度和结果解释成本都更高。它更适合继续作为研究型候选模型保留，而不是在当前阶段直接作为全省最终交付方案。项目最终更重视的是：

- 指标是否稳定领先
- 不同城市上是否一致
- 训练与复现是否简洁

在这三个维度上，`SimpleBiMamba` 更符合最终落地要求。

## 8. 最终方案为什么是“双模型协同”

最终结论不是“某一个模型赢了全部任务”，而是：

- 点预测最优：`SimpleBiMamba + point + log1p`
- 区间预测最优：`SimpleBiMamba + quantile + asinh`

也就是说：

- 一个模型更擅长“报一个最准的数”
- 另一个模型更擅长“报一个合理的范围”

因此，项目最终采用双模型协同方案：

- 点值输出来自点预测模型
- 不确定性区间来自区间预测模型

合并逻辑在 [merge_allcity_predictions.py](C:/Coding/260219_ML-rainfall-time-prediction/scripts/merge_allcity_predictions.py)，输出文件是 [gz_rain_allcities_final_predictions_merged.csv](C:/Coding/260219_ML-rainfall-time-prediction/runs/gz_rain_allcities_final_predictions_merged.csv)。

## 10. 项目最终结论

本项目最终形成了一个较为明确的工程结论：

1. 对贵州省逐日降雨预测任务，`SimpleBiMamba` 是当前阶段更合适的统一骨架。
2. 点预测与区间预测具有不同优化目标，不应强行由一个模型同时承担最优输出。
3. 对点预测，应优先使用 `point + log1p` 方案。
4. 对区间预测，应优先使用 `quantile + asinh + balanced rain weighting` 方案。
5. 在全省 9 个城市上，点预测模型在 `MAE` 上全部优于区间模型，说明“双模型协同”方案具有稳定性。

如果用于实际业务，可以这样理解最终输出：

- 点预测模型负责给出“预计降雨多少毫米”
- 区间预测模型负责给出“降雨大概率落在哪个范围内”

这比强行让一个模型同时承担“单值最准”和“区间最稳”更合理，也更符合降雨预测本身的不确定性特点。
